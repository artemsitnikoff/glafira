## 0. Что мы строим

ATS (Applicant Tracking System) для российского рынка с AI-агентом **«Глафира»**. Ключевая особенность продукта — не воронка (она у всех есть), а то, что AI-агент сам:
- скринит резюме и считает скоринг (0–100),
- ведёт чат-скрининг с кандидатом,
- двигает кандидатов по воронке по правилам,
- делает верификацию по госреестрам (в MVP — мок),
- мониторит нанятых сотрудников после найма (раздел «Пульс»).

**Цель MVP:** рабочая система для ОДНОЙ компании. Архитектура заложена с прицелом на расширение, но без преждевременной мультитенантности.

Источник дизайна: хэндофф из Claude Design (React/JSX-прототипы + 10 md-описаний экранов). Прототипы — это моки; задача — превратить их в production-приложение с реальным API.

## 3. Соглашения API (КРИТИЧНО — соблюдать всем)

### 3.1 Базовый префикс
Все эндпойнты под `/api/v1/`. Пример: `GET /api/v1/vacancies`.

### 3.2 Формат успешного ответа
**Одиночный объект** — отдаём сам объект:
```json
{ "id": "uuid", "name": "Frontend-разработчик", ... }
```

**Список с пагинацией** — конверт:
```json
{
  "items": [ {...}, {...} ],
  "total": 1247,
  "page": 1,
  "page_size": 24,
  "pages": 52
}
```

### 3.3 Формат ошибки (единый для всего API)
HTTP-код + тело:
```json
{
  "error": {
    "code": "VACANCY_NOT_FOUND",
    "message": "Вакансия не найдена",
    "details": null
  }
}
```
- `code` — машиночитаемый SCREAMING_SNAKE_CASE, фронт может на него реагировать.
- `message` — человекочитаемое сообщение на русском (можно показывать пользователю).
- `details` — опционально: массив ошибок валидации полей `[{"field": "salary_from", "message": "..."}]`.

Коды HTTP: `400` нарушение бизнес-правила, `401` не авторизован, `403` нет прав, `404` не найдено, `409` конфликт (дубль), `422` авто-валидация формы запроса (pydantic), `500` внутренняя.

#### 3.3.1 Правило 400 vs 422 (ОБЯЗАТЕЛЬНОЕ соглашение для всех агентов)
Это два разных класса ошибок, они НЕ должны смешиваться — фронт реагирует на них по-разному:

- **`422` — автоматическая валидация формы запроса (pydantic).** Возникает сама, когда тело/параметры не проходят схему (не тот тип, отсутствует required-поле, нарушен паттерн). НЕ генерируется руками. `details` — массив ошибок по полям `[{"field": "...", "message": "..."}]`. **Фронт под 422 подсвечивает конкретные поля формы.**
- **`400` — нарушение бизнес-правила.** Генерируется явно через `ValidationError` (status_code=400) из `app/core/errors.py`, когда запрос синтаксически корректен, но нарушает доменную логику: взаимоисключающие параметры, ЗП выше вилки, переход на недопустимый этап, и т.п. **Фронт под 400 показывает баннер/тост с `message`.**

Иными словами: 422 = «форма заполнена неверно», 400 = «форма верна, но так делать нельзя». Бизнес-исключения ВСЕГДА бросаются через классы из `app/core/errors.py` (`ValidationError`, `NotFoundError`, и т.д.), а не через сырой `HTTPException(detail={...})` — иначе единый exception-handler обернёт detail неправильно и сломает контракт `{error:{code,message,details}}`.

> **Сообщения NotFoundError:** избегать склейки рода («Оценка не найден»). Передавать готовое сообщение целиком ЛИБО хранить шаблон род/число на сущность. (Долг — допустимо доработать позже, но новые сущности оформлять корректно.)

### 3.4 Именование
- **JSON-поля:** `snake_case` (Python-нативно, Pydantic не алиасит). Фронт работает с `snake_case` напрямую — это осознанное решение, чтобы не плодить мапперы.
- **URL:** существительные во множественном числе, kebab при необходимости: `/api/v1/vacancies/{id}/candidates`.
- **Идентификаторы:** все ID — **UUID v4** (строки в JSON). Никаких автоинкрементных int в API.

### 3.5 Пагинация, сортировка, фильтры
- Query-параметры: `?page=1&page_size=24&sort=score&order=desc`.
- Фильтры — отдельные query-параметры: `?city=Москва&score_min=80&stage=interview`.
- `page_size` по умолчанию 24, максимум 100.

### 3.6 Даты и время
- Всё в **UTC**, формат **ISO 8601** с таймзоной: `2026-04-08T14:32:00Z`.
- Фронт форматирует под локаль пользователя (МСК) на клиенте.

### 3.7 Деньги
- Зарплаты — целое число в рублях + код валюты: `{ "salary_from": 200000, "salary_to": 280000, "currency": "RUB" }`.

## 4. Аутентификация и авторизация (MVP)

- **JWT** (access-токен в заголовке `Authorization: Bearer <token>`). Refresh-токен — в HttpOnly cookie.
- Эндпойнты: `POST /api/v1/auth/login`, `POST /api/v1/auth/refresh`, `POST /api/v1/auth/logout`, `GET /api/v1/auth/me`.
- Роли в MVP: `admin`, `recruiter`, `manager` (нанимающий менеджер). Матрица прав — упрощённая, детально в ТЗ по Настройкам. На старте: `recruiter` и `admin` видят всё, `manager` — только свои вакансии (read + оценки).
- Все эндпойнты, кроме `/auth/login`, требуют валидный токен.

## 5. Сквозные требования

### 5.1 company_id (страховка под мультитенантность)
**Все доменные таблицы** имеют колонку `company_id UUID NOT NULL` с дефолтным значением единственной компании. В MVP:
- middleware/зависимость подставляет фиксированный `company_id` из конфига,
- НЕТ изоляции на уровне БД (RLS), НЕТ выбора компании в UI.
- Это просто колонка — чтобы при расширении не делать `ALTER TABLE` на десятках таблиц с боевыми данными.

Реализуется через `CompanyMixin` в `models/base.py`.

### 5.2 152-ФЗ / ПдН-согласия
Согласие на обработку персональных данных — отдельная сущность (`consents`). Правила:
- Кандидат может существовать без подписанного ПдН (например, добавлен из отклика).
- **Верификация по реестрам (Экран 04, таб «Верификация») доступна ТОЛЬКО при подписанном ПдН.** Бек обязан вернуть `403 CONSENT_REQUIRED`, если ПдН нет.
- Согласие имеет номер вида `PD-XXX/26`, дату, канал подписания, статус.
- В UI у кандидата с подписанным ПдН — зелёный бейдж «✓ ПдН».

### 5.3 Аудит-лог
**Каждое изменяющее действие** (POST/PUT/PATCH/DELETE) пишется в таблицу `audit_log`:
- кто (`user_id`), что (`action`), над чем (`entity_type` + `entity_id`), когда (`created_at`), снимок изменений (`changes` JSONB: было/стало для ключевых полей).
- Реализуется middleware + явными вызовами в сервисах для бизнес-событий (перевод кандидата, отказ, найм).
- AI-действия Глафиры тоже логируются с `actor_type = 'ai'`.

### 5.4 Действия Глафиры
Любое действие AI (скоринг, перевод, отказ, отправка сообщения) фиксируется и попадает:
- в аудит-лог (`actor_type = 'ai'`),
- в ленту событий кандидата (таб «Все действия», Экран 04),
- в глобальную ленту событий (Экран 01 «Лента событий · live»).

## 8. docker-compose (состав)

Три сервиса:

## 9. Контракт между беком и фронтом

1. Бек — **источник истины** для контрактов. FastAPI генерит OpenAPI автоматически (`/api/v1/openapi.json`).
2. После любого изменения API бек экспортирует схему в `contracts/openapi.json`.
3. Фронт **может** генерить TS-типы из этой схемы (`openapi-typescript`) ЛИБО вести типы вручную в `src/api/types.ts` — но они ОБЯЗАНЫ соответствовать контракту. В ТЗ-2 я дам и Pydantic-схемы, и зеркальные TS-типы, чтобы оба агента стартовали синхронно.

2. **React-агент:** ТЗ-0 → ждёт готовые эндпойнты из ТЗ-2 → ТЗ-3+ (экраны). Может параллельно поднимать каркас (роутинг, layout, tokens.css, Sidebar) на моках, пока бек не готов.
3. **Ревьювер:** по чек-листам приёмки (отдельный документ к каждому слою).

## 11. Что осознанно НЕ в MVP

- Мультитенантная изоляция (RLS, выбор компании). Только колонка `company_id`.
- WebSocket-realtime. Лента событий и счётчики — **polling** (TanStack Query `refetchInterval`).
- Реальная верификация по госреестрам. Только мок с контрактом.
- Telegram-бот и мессенджеры (WhatsApp/Max/SMS) как каналы. Интерфейс провайдеров заложен, реализация позже.
- Объединение дублей, массовый импорт Excel «в проде» (базовый импорт — да).
- Биллинг (раздел Настроек — UI-заглушка).
