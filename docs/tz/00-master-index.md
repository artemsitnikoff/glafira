# 00. Мастер-индекс ТЗ — «Глафира Рекрутёр» (ATS)

> Точка входа для всех трёх агентов. Прочитать ПЕРВЫМ, затем идти по своему треку.

---

## Что это

Пакет технических заданий на разработку ATS с AI-агентом «Глафира» для российского рынка. Основан на хэндоффе из Claude Design (10 экранов: React/JSX-прототипы + md-описания). Цель MVP — рабочая система для одной компании; архитектура заложена под будущее расширение.

---

## Зафиксированные решения (из обсуждения с продакт-оунером)

| Решение | Значение |
|---|---|
| **Стек бек** | FastAPI + PostgreSQL 16 + SQLAlchemy 2.0 (async) + Alembic + Pydantic v2 |
| **Стек фронт** | React 18 + TypeScript (strict) + Vite + React Router + TanStack Query + Zustand + голый CSS (токены) |
| **Репо** | Монорепо: `/backend` + `/frontend` + `/contracts` |
| **AI «Глафира»** | Реальный Claude API: скоринг резюме (строгий JSON) + чат-скрининг. Верификация по реестрам — мок с финальным контрактом |
| **Источники** | hh.ru + Авито (реальные) + ручной ввод/импорт. Telegram/мессенджеры — интерфейс заложен, реализация позже |
| **152-ФЗ / ПдН** | В схеме БД; блокирует верификацию (403 CONSENT_REQUIRED) |
| **Аудит-лог** | Отдельная таблица + middleware; AI-действия тоже логируются |
| **company_id** | Колонка во всех таблицах (одно значение); НЕ полноценная мультитенантность |
| **Realtime** | Polling (TanStack refetchInterval), НЕ WebSocket |

## Порядок работы (рекомендуемый)

**Фаза 1 — фундамент (параллельно).**
- Бек: ТЗ-0 → ТЗ-1 (поднять схему, миграции, seed) → старт ТЗ-2 (auth + vacancies).
- Фронт: ТЗ-0 → ТЗ-3 (каркас + сайдбар на моках, структура моков = TS-типы из ТЗ-2).

**Фаза 2 — вертикальные срезы (домен бек → экран фронт).**
Рекомендую закрывать сквозными срезами, а не «весь бек, потом весь фронт»:
1. Auth + Sidebar (ТЗ-3) → видна навигация.
2. Vacancies + Funnel (ТЗ-2 ч.1 → ТЗ-5).
3. Candidate detail + Glafira-скоринг (ТЗ-2 ч.2 glafira → ТЗ-6).
4. Candidates pool (ТЗ-7).
5. Pulse (ТЗ-2 pulse → ТЗ-8).
6. Home (ТЗ-2 home → ТЗ-4) — агрегирует данные из всех доменов, удобнее после них.
7. Analytics (ТЗ-9) — последней, нужны накопленные данные.
8. Settings (ТЗ-10) — можно раньше (Глафира-настройки нужны для скоринга/скрининга — сделать `glafira_settings` в Фазе 1).

**Контракт-синхронизация:** после каждого изменения API бек экспортирует `contracts/openapi.json`; фронт сверяет `src/api/types.ts`.

## Критические инварианты (НЕ нарушать)

1. **Верификация без подписанного ПдН → 403 CONSENT_REQUIRED.** Сквозное правило 152-ФЗ.
2. **Каждое изменяющее действие → audit_log** (включая AI с actor_type='ai').
3. **Все доменные записи пишутся с company_id** из контекста.
4. **Скоринг Claude → строгий JSON + защищённый парсинг**, при сбое 502 GLAFIRA_PARSE_ERROR (не 500).
5. **→ hired создаёт сотрудника в Пульсе** + Глафира генерит план адаптации.
6. **Единый формат ошибок** `{error:{code,message,details}}` и ответов (объект / `{items,total,...}`).
7. **JSON-поля snake_case**, фронт работает напрямую (без camelCase-мапперов).
8. **Прототипы из Claude Design — источник пиксель-перфекта.** При расхождении значений с tokens.css берём из прототипа, обновляем токены.
